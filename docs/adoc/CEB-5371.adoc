= Endpoint tokenization/sendPassCode
:toc:
:toclevels: 3

== Objetivo
Documentar el contrato HTTP del endpoint `POST /{tenant}/{customerType}/tokenization/sendPassCode` con ejemplos de request y response basados en la implementacion actual.

== Referencia de flujo
Este endpoint tiene dos vistas complementarias de flujo:

- Secuencia end-to-end: `docs/diagram/sequence/sendPassCode-tokenization-e2e.puml`
- Diagrama de actividad (flujo puro): `docs/diagram/sequence/sendPassCode-tokenization-activity.puml`

== Endpoint
[source,http]
----
POST /{tenant}/{customerType}/tokenization/sendPassCode
Content-Type: application/json
Accept: application/json
----

== Path params
[cols="1,3",options="header"]
|===
|Parametro |Descripcion
|`tenant` |Identificador del banco/tenant (usado para configuracion y llaves criptograficas).
|`customerType` |Tipo de cliente requerido por el routing base del controlador.
|===

== Request body
El body corresponde a la entidad `SendPassCodeTokenization`.

[cols="1,1,3",options="header"]
|===
|Campo |Tipo |Descripcion
|`tokenRequestorID` |string |Identificador del token requestor.
|`tokenReferenceID` |string |Referencia del token.
|`panReferenceID` |string |Referencia del PAN.
|`clientWalletAccountID` |string |Cuenta wallet del cliente.
|`otpMaxReached` |string |Indicador/estado de maximos intentos OTP.
|`otpMethodIdentifier` |string |Canal/metodo OTP.
|`otpValue` |string |OTP a notificar.
|`otpDestiny` |string |Destino OTP (email o telefono). Si es email, el endpoint enruta por canal email; de lo contrario por sms.
|`otpReason` |string |Motivo OTP.
|`otpExpirationDate` |string |Fecha de expiracion OTP.
|`deviceInfo` |object |Objeto con metadata del dispositivo (se usa al persistir log).
|`encryptedData` |string |Payload cifrado/firmado que al desencriptar debe mapear a `EncryptedDataSendPass`.
|===

== Contenido esperado dentro de `encryptedData` (desencriptado)
Internamente el controlador hace `verify` + `decrypt`, y el JSON resultante se mapea a `EncryptedDataSendPass`.

[source,json]
----
{
  "cardholderInfo": {
    "primaryAccountNumber": "4111111111111111",
    "name": "JOHN DOE",
    "otpDestiny": null,
    "highValueCustomer": false
  },
  "riskInformation": {
    "accountHolderName": "JOHN DOE",
    "visaTokenScore": "05",
    "visaTokenDecisioning": "APPROVE"
  },
  "tokenInfo": {
    "token": "4761739001010010",
    "tokenType": "NETWORK",
    "tokenStatus": "ACTIVE",
    "tokenAssuranceMethod": "RISK_BASED_AUTH",
    "numberOfActiveTokensForPAN": 1,
    "numberOfInactiveTokensForPAN": 0,
    "numberOfSuspendedTokensForPAN": 0,
    "autoFillIndicator": true
  }
}
----

== Responses
=== 200 OK (flujo normal)
Cuando no hay excepcion, la respuesta es `NovoResponse` serializada.

[source,json]
----
{
  "rc": "0",
  "msg": "OK",
  "data": "Notification category: tkSendPassCode Activate"
}
----

Notas:

- Si la categoria `tkSendPassCode` esta inactiva, `data` cambia a `Notification category: tkSendPassCode Disabled`.
- Si `notificationCategories.emulate=true`, se devuelve 200 con `rc/msg` y sin ejecutar envio/persistencia (normalmente `data` ausente).

=== 200 OK (body invalido tras decrypt)
Si falla el mapeo del JSON desencriptado a `EncryptedDataSendPass`, el controlador retorna `HTTP 200` sin body.

=== Error de negocio (`NovoException`)
Si se lanza `NovoException` (por ejemplo, cliente no encontrado), el endpoint responde con el HTTP code asociado a `ResponseType` y el body de error.

Ejemplo referencial de estructura:

[source,json]
----
{
  "rc": "<codigo_negocio>",
  "msg": "<descripcion>"
}
----

=== 500 Internal Server Error
Ante excepcion no controlada:

[source,text]
----
<mensaje de la excepcion>
----
